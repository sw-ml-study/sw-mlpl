1) Dependency flow ambiguity

Treat mlpl-parser and mlpl-array as peers.

Final rule
mlpl-core → shared low-level types only
mlpl-parser depends on mlpl-core
mlpl-array depends on mlpl-core
mlpl-runtime depends on mlpl-core + mlpl-array
mlpl-eval depends on mlpl-core + mlpl-parser + mlpl-runtime + mlpl-array
Why

Parser should produce syntax/AST, not semantic tensor objects.
So array literals in the AST should be represented as syntax nodes like:

scalar literal
array literal
list of expressions
maybe shape annotations later

They should not use mlpl-array::Shape directly.

That keeps parser context small and avoids premature semantic coupling.

Practical guidance

If a parser task needs to represent an array literal, use parser-owned AST nodes, not runtime/array types.

Use contract directories first, not workspace crates.

Final rule
