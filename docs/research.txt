APL2 (developed by IBM in the 1980s) is best understood as a major evolution of classic APL, aimed at making the language more practical for large-scale programming, data processing, and system integration.

High-level decision
1) Why not make classic APL or GNU APL your main target
GNU APL is valuable as a reference point because it implements ISO APL Extended and gives you a concrete open implementation to compare against. 

But as the center of your teaching/demo project, classic APL has drawbacks:

the historical semantics are not always the cleanest teaching story

the notation is iconic but harder to make approachable

the implementation story is less obviously “modern Rust architecture for a new runtime”

So I would use GNU APL as a compatibility oracle / comparison baseline, not as the core design target. GNU APL is free software from GNU, and GNU APL ecosystem materials emphasize GPL/LGPL compatibility for associated libraries. 

2) Why not make Dyalog your main target
Dyalog is extremely important conceptually because it carries forward many of the APL2-style practical features you care about, including namespaces and broader production-oriented tooling. Dyalog’s docs explicitly describe namespaces as analogous to nested workspaces. 

But Dyalog itself is not open source; it is free for non-commercial use under a Basic Licence, but it is not free software. 

So Dyalog is best used as:

a feature reference

a semantics benchmark

an example of what “APL2 evolved toward”

not as the implementation you center your Rust-teaching repos around.

3) Why not make J your main target
J is mature, portable, actively maintained, and its source is available under GPLv3, with J Software also offering commercial licensing. The project site says the source is provided under both commercial and GPL 3 licenses, and the current release line includes J9.6. 

Technically, J is excellent. It also gives you:

ASCII syntax

strong array semantics

rank/cell thinking

tacit programming

modularity through locales/namespaces-like mechanisms

Its docs emphasize rank, cells, forks, and composition, and J’s own papers position it as essentially a superset of APL in spirit. 

But for your specific teaching/demo goals, J has two problems:

the tacit / fork-heavy style can become visually and cognitively dense very quickly

it is less naturally aligned with your desire to demonstrate APL2-style improvements like nested structure in an intuitive, visual way

So I would absolutely study J, but mostly to learn:

rank-oriented execution

composition patterns

interpreter/runtime organization ideas

not as the main language you build demos around.

4) Why BQN is the best primary target
BQN is the best fit because it gives you a modern, teachable reinterpretation of APL-family ideas while still being close enough to the tradition to let you explain what changed and why.

The BQN docs are especially useful for your project because they explicitly discuss:

array model

types

shape / rank / depth

the relationship between BQN’s model and the nested array model of APL2, Dyalog, GNU APL, and NARS. 

That last part matters a lot: BQN’s own design commentary gives you a built-in narrative for blog posts and videos:

“Here is the APL2-style nested array tradition, and here is a modern alternative model.”

That is gold for teaching.

BQN is also open source. The main BQN repo is public, and at least one prominent implementation, dzaima/BQN, is MIT licensed. 

Why BQN maps well to your architecture

How to keep agents from wandering the repo
There are several practical tactics.

1. Start each agent in a subdirectory
When possible, initialize an agent in:

crates/mlpl-array/

or contracts/array-contract/

not at repo root.

That alone reduces accidental context acquisition.

2. Give each agent a local task packet
Each packet should include only:

task statement

allowed directories

required tests

prohibited dependencies

escalation triggers

Example:

Task: Implement reshape validation for mlpl-array.
Allowed dirs:
  - crates/mlpl-array
  - contracts/array-contract
Do not inspect:
  - crates/mlpl-eval
  - apps/*
Success:
  - reshape_contract.rs passes
  - no API changes outside mlpl-array
Escalate if:
  - contract seems inconsistent
  - new error type required in mlpl-core
This is much better than dumping architecture docs into every task.

3. Make “upstream only” the default read rule
An agent may read:

its own crate

contract crate

direct upstream crates only if necessary

An agent may not read:

downstream UI

unrelated demos

future milestones

blog/docs unless specifically needed

That keeps mental load low.

4. Prefer narrow facade crates and summary types
For example, mlpl-trace should receive value summaries, not whole runtime internals.

Bad:

trace agent must understand full Value internals

Better:

runtime exposes ValueSummary

trace consumes ValueSummary

This reduces coupling and context size.

Recommended crate API style for low-context agents
To make this work, APIs need to be intentionally narrow.

Prefer:
small structs

small enums

pure functions

no deep hidden side effects

explicit error types

explicit conversion/adaptation layers

Avoid:
giant “god” modules

big prelude exports

deep macro magic early

broad trait hierarchies too soon

runtime globals

Those increase the amount of context an agent must hold.

Recommended “capsule” design per crate
Each crate should behave like a capsule.

Each capsule should have:
1. README.md
One-screen overview.

2. AGENTS.md
Tasking and local rules.

3. src/lib.rs
Exports only the public API, with minimal surface.

4. src/
Small modules only.

5. tests/
Local tests.

6. examples/
Tiny examples.

7. CHANGELOG.md or MILESTONE.md
Local progress.

That way an agent can operate inside a capsule without browsing the world.

# MLPL Agent Rules

## Mission
MLPL is a Rust-first tensor and array programming language with visual debugging and ML experimentation support.

## Development model
Work is task-based and contract-first.

Default rule:
- one task
- one branch
- one worktree
- one PR

## Context containment
Agents should begin with only:
- their task packet
- their local crate directory
- their contract directory
- their local crate `AGENTS.md`

Do not browse the wider repo unless required by escalation.

## Allowed read scope
1. local crate
2. local contract
3. direct upstream public API, only if needed
4. coordinator-provided summary
5. selectively revealed external files

## Forbidden behavior
- do not refactor unrelated crates
- do not edit root workspace files unless task explicitly allows it
- do not widen public APIs casually
- do not browse apps/UI/docs for implementation inspiration unless task requires it
- do not solve adjacent problems “while here”

## Dependency rule
Prefer one-way dependencies:
core → array/parser → runtime → eval → trace → viz/wasm/apps → ml

## PR rule
Open a small PR when local acceptance tests pass.
Use task ID in branch, PR title, and final squash commit.

## Escalation
Escalate when:
- contract is ambiguous
- new upstream API is needed
- forbidden file would need modification
- local implementation reveals architecture conflict

8) Suggested Git workflow commands
Use the same sequence every time.

8.1 Create worktree for a task
From your main checkout:

git fetch origin
git checkout main
git pull --ff-only origin main

git worktree add ../mlpl-worktrees/wt-array-reshape-002 \
  -b feat/array/array-reshape-002-reshape-op \
  origin/main
Then hand the agent the worktree path.

8.2 Agent local workflow
Inside the worktree:

cargo test -p mlpl-array
cargo test -p array-contract
cargo fmt
cargo clippy -p mlpl-array --all-targets -- -D warnings
git status
git add .
git commit -m "ARRAY-RESHAPE-002: implement reshape validation and operation"
git push -u origin feat/array/array-reshape-002-reshape-op
Then open PR.

8.3 Integration workflow
Integration agent does:

git fetch origin
git checkout main
git pull --ff-only origin main

# optional: inspect PR branch in temporary worktree
git worktree add ../mlpl-worktrees/wt-review-array-reshape-002 origin/feat/array/array-reshape-002-reshape-op
Run broader checks if needed, then squash merge PR.

8.4 Cleanup
After merge:

git worktree remove ../mlpl-worktrees/wt-array-reshape-002
git branch -d feat/array/array-reshape-002-reshape-op
git push origin --delete feat/array/array-reshape-002-reshape-op
9) Recommended merge policy
Use squash merge by default
Why:

one task becomes one clean commit on main

intermediate agent commits do not clutter history

task history remains easy to browse

easier to revert whole tasks if needed

Exception
Use normal merge only when you intentionally want to preserve internal branch history for a large multi-step epic.

For MLPL, I would keep the default as squash.

10) How to reduce conflicts even more
Git mechanics matter, but task design matters more.

Best conflict reducers
Keep shared root files coordinator-owned
These should usually be touched only by coordinator or merge agent:

root Cargo.toml

Cargo.lock

top-level docs

CI config

workspace metadata

Prefer crate-local manifests when possible
If adding a dependency only changes a crate-local Cargo.toml, conflict risk is much lower.

Avoid simultaneous contract edits
Try not to have two agents editing the same contract directory at once.

Sequence upstream before downstream
Merge mlpl-array tasks before spawning dependent mlpl-runtime tasks if possible.

Keep tasks short-lived
The fastest conflict reducer is fast merge turnaround.

# MLPL Coordinator Runbook

This repository is `sw-mlpl`.
The product, language, and platform described in docs and README are called **MLPL**.

## Purpose

This file defines the operating procedure for coordinating multiple AI coding agents working in parallel on MLPL.

The goals are:

- low friction
- low interference
- narrow task scope
- contract-first development
- short-lived branches
- reproducible integration
- minimal agent context

---

## Core Rule

Every implementation task should follow this pattern:

- one task
- one branch
- one worktree
- one PR

Default merge style: **squash merge**.

---

## Repository Model

The repository is a **cellular monorepo**.

Each crate is a capsule with:
- its own `README.md`
- its own `AGENTS.md`
- its own tests
- narrow dependencies
- a matching contract area in `contracts/`

Agents should work locally in one capsule plus one contract directory.

---

## Naming Rules

### Repo
- Git repo name: `sw-mlpl`
- Language/platform name in docs: `MLPL`

### Task IDs
Format:
`<AREA>-<TOPIC>-<NNN>`

Examples:
- `ARRAY-SHAPE-001`
- `PARSER-LEX-001`
- `TRACE-EVENT-001`

### Branch names
Format:
`feat/<area>/<task-id-lower>-<slug>`
`fix/<area>/<task-id-lower>-<slug>`
`docs/<area>/<task-id-lower>-<slug>`
`spike/<area>/<task-id-lower>-<slug>`

Examples:
- `feat/array/array-shape-001-shape-type`
- `feat/parser/parser-lex-001-basic-lexer`

### Worktree names
Format:
`wt-<task-id-lower>`

Examples:
- `wt-array-shape-001`
- `wt-parser-lex-001`

### PR titles
Format:
`[<area>] <TASK-ID>: <summary>`

Examples:
- `[array] ARRAY-SHAPE-001: add Shape type and invariants`
- `[parser] PARSER-LEX-001: implement numeric and punctuation lexer`

---

### Create a task branch and worktree

```bash

the assigned worktree

the assigned crate directory

the assigned contract directory

Task Lifecycle
1. Ready
Task file exists in TASKS/ready/.

2. Active
Task file is moved to TASKS/active/.
Branch and worktree are created.
Agent begins implementation.

3. Blocked
If blocked, move task file to TASKS/blocked/ or keep in active/ with a blocked note.

4. PR Open
PR references the task ID and contract.

5. Done
After merge:

move task file to TASKS/done/

update WORKTREES.md

remove worktree

delete branch

Context Containment Rules
Agents start with only:

the task packet

the local crate AGENTS.md

the local contract README/tests

the allowed directories listed in the task

Agents should not read:

unrelated crates

apps

top-level docs

downstream consumers
unless the task explicitly allows it or escalation approves it.

Escalation Ladder
Level 0 — local only
Allowed:

assigned crate
