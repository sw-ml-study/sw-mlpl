My recommendation: organize MLPL feature requests into paper-driven capability buckets
Instead of treating each paper as a one-off, I would group them into a handful of reusable capability families.

1. Architecture research primitives
Use-cases: mHC, Engram

Needed MLPL capabilities:

tensor and block-structured linear algebra

constrained parameterization primitives

graph-like model topology description

pluggable residual / routing / memory modules

per-layer instrumentation hooks

attention-vs-memory ablation support

visualization for topology, activations, retrieval hit-rates, long-context behavior

Why this matters:
These papers are really about changing the computation graph itself, so MLPL needs to make architecture variation first-class rather than burying it in handwritten Rust/CUDA glue. 

2. Inference-time adaptation and reward-loop primitives
Use-cases: ICRL, Tool-use ICRL, RelayLLM

Needed MLPL capabilities:

trajectory records

reward annotation streams

prompt/context accumulation operators

tool schemas and tool-call execution

controller/worker model orchestration

token-level or step-level routing

eval harnesses for cost, latency, success rate, and reward over time

Why this matters:
These papers emphasize test-time learning or coordination, not just offline training. Your language will benefit from native constructs for episodes, actions, rewards, tool calls, and replay. 

3. Fine-tuning and optimizer experiment primitives
Use-cases: Learning Rate Matters, plus LoRA/QLoRA/shared-subspace work

Needed MLPL capabilities:

parameter-efficient adapter declarations

hyperparameter sweep syntax

optimizer/learning-rate schedule abstractions

rank, init, precision, quantization controls

experiment registry and comparison plots

Hessian/eigenvalue or curvature probe hooks

Why this matters:
A lot of papers are not about inventing a new layer; they are about showing that results depend on training protocol details. MLPL should make these protocols easy to vary and inspect. 

4. Agentic code-generation / systems optimization primitives
Use-cases: CUDA Agent, EVA, Astrolabe

Needed MLPL capabilities:

task → code → verify → profile → revise loops

external tool invocation

benchmark and reward extraction

sandboxed execution

multi-stage training pipelines

rubric/reward composition

artifact generation: charts, docs, demos, slides

Why this matters:
This is where MLPL becomes both an ML language and an ML systems laboratory. The same substrate can drive model training experiments and code-generating agents that build kernels, demos, and benchmarks.
